Here's a summary of every major design decision made and why.

**Type hierarchy.** Registers are split into five named sub-types (`gpr64`, `gpr32`, `gpr16`, `gpr8l`, `gpr8h`) and then unified under a single `gpr` sum type. This means the instruction selector can express "this operand must be a 64-bit register" precisely â€” mismatches are caught at the type level rather than at encode time. The same principle applies to memory operands: `mem` carries an `isize` and `fmem` carries an `fsize`, so you can never accidentally use a float memory reference where an integer reference is expected.

**Two separate float sub-systems.** `x87_instr` and `sse_scalar_instr` are completely separate variant groups. A backend that emits only SSE2 scalar code (the norm on modern compilers) can ignore `x87_instr` entirely, and vice versa. SSE2 scalar uses `xmm_reg` and `sse_operand`; x87 uses `st_reg` and `fp_operand`.

**The MOVSD / CMPSD name clash.** x86 happens to reuse the same mnemonics for two completely unrelated instructions: the string-move-dword (`MOVSD` = move `[RSI]â†’[RDI]`) and the SSE scalar-double move. In the ASDL module, constructor names must be globally unique, so the string variants were renamed `MOVS_DWORD` and `CMPS_DWORD` with comments noting the assembler mnemonic they emit.

**LOCK modelled structurally, not as a prefix.** `LOCK` is only legal on about a dozen specific instructions (ADD, XCHG, CMPXCHG, etc.). Putting it in `prefix` (like `REP`) would allow it to be attached to any instruction, which is wrong. Instead it is modelled as `LOCK(instr body)` inside `atomic_instr`, making the constraint explicit â€” the type of `body` can be further restricted by the code generator.

**`RepPrefix` vs `REP`.** `REP` is a variant of `rep_kind` (the flavour of repeat: `REP`, `REPE`, `REPNE`). Using `REP` again as the `prefix` constructor name would be a duplicate. The prefix constructor is therefore named `RepPrefix(rep_kind k)`.

**Pseudo-instructions inline.** `ILabel(label)` and `IAlign(int)` sit alongside real instructions in the `instr` sum type. This avoids a separate "instruction-or-directive" union and lets linear passes handle labels and alignment naturally.

**Code structure reflects real compiler output.** The `block` type has an explicit `terminator` field typed as `cf_instr` â€” so the type system enforces that every basic block ends with exactly one control-flow instruction. `func` carries calling-convention, callee-saved registers (both GPR and XMM for Windows x64), frame size, and frame-pointer flag â€” everything the prologue/epilogue emitter needs.
